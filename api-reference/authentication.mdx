---
title: "Аутентификация (API)"
description: "Эндпоинты входа, обновления токена и профиля."
---

Аутентификация строится на **JWT** (SimpleJWT). Основные маршруты:

- `POST /v1/auth/login/` — выдача пары **access** / **refresh**
- `POST /v1/auth/refresh/` — новый **access** по **refresh**
- `POST /v1/auth/logout/` — инвалидация **refresh** (blacklist)
- `GET` / `PUT` / `PATCH /v1/auth/profile/` — профиль пользователя

В спецификации OpenAPI для схем безопасности также указаны **Basic** и **cookie** (`sessionid`); для интеграций по HTTP API обычно используется заголовок **`Authorization: Bearer`** с access-токеном.

<Note>
  Кроме публичных эндпоинтов входа и сброса пароля, защищённые методы требуют валидного JWT.
</Note>

## POST /v1/auth/login/

### Тело запроса

<ParamField body="username" type="string" required>
  Имя пользователя или email (как в вашей учётной записи).
</ParamField>

<ParamField body="password" type="string" required>
  Пароль.
</ParamField>

### Пример

```bash
curl --request POST \
  --url https://api.yume.cloud/v1/auth/login/ \
  --header 'Content-Type: application/json' \
  --header 'Accept: application/json' \
  --data '{
    "username": "you@example.com",
    "password": "your_password"
  }'
```

### Ответ

В ответе — JWT access и refresh (см. фактическое тело в вашей среде и схемы в OpenAPI).

```json
{
  "access": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refresh": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
}
```

## POST /v1/auth/refresh/

Передайте **`refresh`** в теле JSON — в ответе придёт новый `access` (и при необходимости обновлённый `refresh`).

## Использование токена

```
Authorization: Bearer <access>
```

<Warning>
  Не передавайте токены в URL и в публичных репозиториях.
</Warning>
