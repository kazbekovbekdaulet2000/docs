Ниже — минимальный сценарий: вход по JWT и чтение списка клиентов CRM. Все пути начинаются с **`/v1/`**.

## Что нужно

- Учётная запись Yume Cloud с логином и паролем (уточните у администратора).
- `curl` или другой HTTP-клиент.

<Note>
  Запросы отправляйте на **`https://api.yume.cloud`** только по **HTTPS**.
</Note>

---

<Steps>
  <Step title="Вход и получение JWT">
    Отправьте учётные данные на `POST /v1/auth/login/`.

    <CodeGroup>

    ```bash cURL
    curl -X POST https://api.yume.cloud/v1/auth/login/ \
      -H "Content-Type: application/json" \
      -d '{"username": "your@email.com", "password": "your_password"}'
    ```

    ```json Ответ
    {
      "access": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
      "refresh": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
    }
    ```

    </CodeGroup>

    Скопируйте значение **`access`** — его передаёте как `Bearer` в следующих шагах.
  </Step>

  <Step title="Первый защищённый запрос">
    Запросите страницу списка клиентов CRM (метод и схема ответа — в OpenAPI).

    <CodeGroup>

    ```bash cURL
    curl 'https://api.yume.cloud/v1/crm/clients/?page=1&pageSize=20' \
      -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..." \
      -H "Accept: application/json"
    ```

    ```json Пример ответа (фрагмент)
    {
      "results": [],
      "count": 0,
      "next": null,
      "previous": null
    }
    ```

    </CodeGroup>

    Точная форма пагинации и полей зависит от версии API и настроек бэкенда — ориентируйтесь на спецификацию и фактический JSON.

    <Tip>
      Для списков в OpenAPI обычно используются параметры **`page`** и **`pageSize`**.
    </Tip>
  </Step>

  <Step title="Обновление токена">
    Когда `access` перестанет приниматься, обновите его через `POST /v1/auth/refresh/` с телом `{"refresh": "..."}`.
  </Step>
