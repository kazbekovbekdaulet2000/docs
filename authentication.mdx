---
title: "Аутентификация"
description: "Как аутентифицировать запросы к API Yume Cloud."
---

API использует **JWT** (SimpleJWT). Вы отправляете логин и пароль на `POST /v1/auth/login/` и получаете **`access`** и **`refresh`**. Во всех защищённых запросах передаётся заголовок **`Authorization: Bearer &lt;access&gt;`**.

<Note>
  Без валидного access-токена защищённые методы возвращают `401 Unauthorized`.
</Note>

## Получение токенов

```bash
curl -X POST https://api.yume.cloud/v1/auth/login/ \
  -H "Content-Type: application/json" \
  -d '{"username": "you@example.com", "password": "your_password"}'
```

Успешный ответ содержит пару JWT (поля в стиле SimpleJWT):

```json
{
  "access": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refresh": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
}
```

Сохраните **`access`** для вызовов API и **`refresh`** для обновления сессии.

## Использование access-токена

```bash
curl https://api.yume.cloud/v1/crm/clients/?page=1&pageSize=20 \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIs..."
```

Формат заголовка:

```
Authorization: Bearer <access>
```

<Warning>
  Не публикуйте токены в клиентском коде, репозиториях и URL. Храните их в переменных окружения или в менеджере секретов.
</Warning>

## Обновление access

Когда `access` истёк, отправьте **`refresh`** на `POST /v1/auth/refresh/`:

```bash
curl -X POST https://api.yume.cloud/v1/auth/refresh/ \
  -H "Content-Type: application/json" \
  -d '{"refresh": "<your_refresh_token>"}'
```

Ответ снова содержит актуальные токены (см. схему `TokenRefresh` в OpenAPI).

## Выход

`POST /v1/auth/logout/` с телом, содержащим `refresh`, добавляет refresh в чёрный список (при включённом `token_blacklist` на стороне сервера).

## Типичные ошибки

| Код | Смысл | Что сделать |
|-----|--------|-------------|
| `401` | Нет токена, неверный или истёкший access | Повторить вход или вызвать `/v1/auth/refresh/` |
| `403` | Токен есть, но недостаточно прав | Проверить роли и доступы в аккаунте |

## Рекомендации по безопасности

- Передавайте токен только в заголовке `Authorization`, не в query-параметрах.
- Храните учётные данные и токены вне исходного кода.
- При компрометации смените пароль и завершите сессии через API или интерфейс продукта.
